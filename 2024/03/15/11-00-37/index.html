介绍MTR课表

总体介绍

MTR课表是我在去年和组里同学一起开发的项目，初衷是帮南理工的同学们查询课表、成绩等教务信息。主要的技术栈是springboot、mybatis-plus、redis、mysql、es做搜索、kafka做消息队列。

背景

做这个是因为我们学校的教务信息查看很麻烦，手机上只能打开浏览器，进入学校网站才能查看。所以就想着开发一下这个小程序，方便同学们查询这些信息。

我的模块

我主要负责登陆模块和论坛模块。

如何获取教务信息

学校的教务网站有个统一的入口，通过这个入口就可以访问各种信息，课表、考试教室、考试成绩等等，具体实现是我们可以登录教务系统拿到一个cookie，然后之后就可以直接带着cookie访问来查询这些信息了。

登陆模块

第一个版本使用的是最常规的Session的登录方式（session-localStorage方案），后来改了多服务器以后，Session这种方式就出现了问题，首先就是Session不共享的问题。

我们最后选择了JWT，解决了Session共享的问题，用户登陆时，服务器端生成一个token令牌响应给前端。后续的请求中需要携带令牌到服务器端，服务器端可以解析token，得到用户信息做校验。

jwt是目前最流行的跨域认证的解决方案，是基于token的认证授权机制。本质是一组字符串，由3个部分组成。

CSRF攻击

跨站请求伪造，通常发生在受害者已经登录了一个网站时，攻击者通过欺骗方式诱导受害者在其他网站上执行操作，从而利用受害者在原始网站上的身份进行操作。用jwt可以避免CSRF攻击，因为在发起请求时，cookie会自动把token请求出去，而jwt不会。

jwt优点

  • 不用服务器存储，节省服务器资源
  • 避免CSRF攻击，由前端保存在localstorage中
  • 集群可用

评论实现方式

把评论和回复都当做评论，按时间排序。因为我们回复评论的情况较少，实现起来也比较简单。

多线程评论数量不同步

tomcat是多线程的，所以单机也可能会出现同步问题。一开始使用sycronized锁实现，不同管锁的获取和释放，挺方便的。后来因为使用了多个服务器，单机锁不适用了（集群环境下由于JVM不同，内部的锁监视器也不同）。

后来想着使用乐观锁的方式来实现也可以，利用mysql自己的行锁写锁，在更新操作的时候在where上多加一个条件 (更新之前先查询，记录下查询到的值，在where比较，不同说明中间被修改过)，如果更新失败，就睡一下，然后重试。这种实现方式在高并发的时候效率并不高，因为如果出现同步问题的时候，锁是在数据库的，相当于在磁盘上，效率并不高。

后来抱着学习的心态使用了redis的分布式锁，原本打算用setnx做一个简单的分布式锁。考虑到可能出现客户端宕机，没有释放锁，导致死锁。于是加了过期时间。

但是如果过期时间到了，程序还没执行完呢？万一把别人的锁释放了，可以使用uuid标识，防止释放别人的锁。没有超时续约，使用redission，watchDog机制优雅续期。

为什么不直接用redis统计评论数？

  • 因为想要数据直接持久化在数据库里，redis只作为工具。

单机为什么不用事务呢？

  • 事务不能解决并发问题啊，事务只是保证事务不出错，而不是不执行。

Nginx限流

令牌桶算法：

令牌以固定速率产生，缓存到令牌桶里面。令牌桶放满时，多余的令牌被丢弃。每次请求要消耗等比例的令牌才能被处理。令牌不够的时候，请求被缓存。允许某种程度的突发处理（可能有一段时间执行的快）。

漏桶算法：

缓存请求（请求放进桶里），然后匀速处理请求（处理漏出来的水），请求满了，就丢弃（水溢出桶）。对突发容量不做额外处理（永远保持匀速）

打包步骤

  2. mvn install
  3. 在target里找到jar包
  4. Java -jar xx.jar运行，可以用nohup 后台运行

排查cbase空指针异常

  • 工作：com.sankuai.meituan.waimai.sc.framework.executor.impl.QueuedGraphExecutor.buildContext(QueuedGraphExecutor.java:96) 抛出空指针异常，具体语句是graphConfig.getActionConfigMap(), 经排查发现是graphConfig为空导致的，这个对象是由GraphManager 类监听ContextRefreshedEvent 事件后触发构造的。即在调用执行流时，底层的初始化还没好。api的节点部署完成后过一段时间才会被发现，执行流已经加载好了；而cbase节点一部署完成就会被发现，执行流的相关参数还没构造好，调用时对象为空。
  • 结果与收获：通过MTthrift自定义状态尝试解决，在服务端启动时设置其运行状态为未启动状态，服务预热后将状态设置为正常，此时开始接收流量，从而达到延迟接收流量的效果。通过本次错误排查，了解了服务底层初始化的过程，加深了对ContextRefreshedEvent事件及其在应用启动过程中的作用的理解。

空指针异常

原因：下游服务部署完成后偶发空指针异常错误，报错的地方来自底层执行框架的GraphManager类 ，执行框架监听了contextRefreshedEvent事件，会去读取执行流描述文件，将其加载到一个变量中去。后面从这个变量中获取到执行流的执行链路。空指针异常就发生变量为空，原因是服务接受流量的时机过早，底层执行框架还没加载完成。

解决：使用MTthrift提供的自定义状态，在一开始把服务设置成关闭状态，使用ApplicationReadyEvent事件把服务设置成激活状态，该事件的发生在contextRefreshedEvent之后（Spring中事件的触发默认是在主线程按顺序执行的）。

可以把事件提前吗？执行流初始化事件放到容器准备好之前，容器准备好之前服务不会启动，可以使用ApplicationStartingEvent。

事件介绍

SpringApplicationEvent

  • ApplicationStartingEvent：在environment准备之前，非常早期的事件，此时Spring还没有开始读取任何配置以及初始化日志
  • ApplicationEnvironmentPreparedEvent：environment已经准备好，在context创建前
  • ApplicationPreparedEvent：bean definition 已经加载，但是没有refresh
  • ApplicationStartedEvent：context已经被refresh，但application和command-line的runner还没被调用
  • ApplicationReadyEvent：application和command-line的runner都被调用后触发
  • ApplicationFailedEvent：启动失败触发

ApplicationContextEvent

  • ContextRefreshedEvent：当spring容器context刷新时触发
  • ContextStartedEvent：当spring容器context启动后触发
  • ContextStoppedEvent：当spring容器context停止时触发
  • ContextClosedEvent：当spring容器context关闭时触发，容器被关闭时，其管理的所有单例Bean都被销毁。

ApplicationStartingEvent、ApplicationEnvironmentPreparedEvent、ApplicationPreparedEvent三个事件开始较早，不能通过component注解扫描的形式去捕获，可以在spring.factories 文件中注册监听器

执行顺序

  • ApplicationStartingEvent - 应用启动，最早的事件。
  • ApplicationEnvironmentPreparedEvent - 环境准备好，ApplicationContext 尚未创建。
  • ApplicationPreparedEvent - ApplicationContext 创建完成，但未完全初始化。
  • ContextRefreshedEvent - ApplicationContext 完全刷新，所有 Bean 初始化完成。
  • ApplicationStartedEvent - 应用已启动，ApplicationRunner 和 CommandLineRunner 还未运行。
  • ApplicationReadyEvent - 应用完全准备好，ApplicationRunner 和 CommandLineRunner 运行完毕。
  • ContextStartedEvent - ApplicationContext 手动启动时触发（可选）。
  • ContextStoppedEvent - ApplicationContext 手动停止时触发（可选）。
  • ContextClosedEvent - ApplicationContext 关闭时触发，应用关闭。
  • ApplicationFailedEvent - 应用启动失败时触发。

MO周期心跳包阻塞

现象：定时100ms的周期心跳包阻塞到700ms

排查：

  • 先排查网络问题，把无线换成有线

  • 数据包问题(中间跳过了几个包)，接收数据排查，校验和错误或长度错误

  • 依次删除代码中其他模块，问题仍未解决

  • 只剩udp后还存在问题，考虑是udp底层问题，写日志，看日志是否阻塞

  • top 查看cpu，是否异常占用

  • 发现Linux日志管理systemd-journald，每次阻塞时cpu占用都很高(80以上)

  • 查看内核日志，发现有频繁的写日志行为(微妙级)，日志文件很大

  • 后来联系厂家，发现是有一个硬件接口有问题，导致系统日志频繁读写，导致的阻塞

  • 禁用systemd-journald服务后解决。
