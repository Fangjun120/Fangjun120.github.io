synchronized锁升级

展开查看

  • 锁主要有四种状态，无锁、偏向锁、轻量锁、重量锁，随着竞争的激烈逐渐升级，不可降级。

    • 偏向锁，会记录获取锁的线程ID，在没有别的线程竞争的时候，一直偏向当前线程，当前线程就可以一直进入synchronized修饰的代码块一直运行。
    • 轻量锁，其他线程以自旋方式尝试获取锁
    • 重量锁，赖操作系统底层的Muptex Lock，其他线程不会自旋 (cpu效率下降)，而是阻塞。

  • 共享资源首次被访问时，加偏向锁，程序间串行访问代码；出现竞争时，JVM先尝试轻量级锁，以CAS自旋方式获取锁，成功则状态为轻量锁，失败就升级到重量锁；

  • 如果共享资源锁已经被某个线程持有，此时是偏向锁状态，未释放锁前，再有其他线程来竞争时，则会升级到重量级锁，另外轻量级锁状态多线程竞争锁时，也会升级到重量级锁

展开查看 IO模型如下，两个进程想跨网络通信，先把数据从用户空间拷贝到缓冲区，再拷贝到内核缓冲区，再写到设备如网卡。

  • 同步阻塞BIO会一直阻塞等待数据、同步非阻塞NIO在等待时间内轮询、异步非阻塞AIO

可重入锁的实现原理

展开查看

为每个锁维护一个拥有者线程标识和计数器，在加锁和解锁操作时对计数器进行递增和递减，并通过并发控制确保线程对锁状态的访问是安全的。

如ReentrantLock，在内部维护了一个锁状态变量和一个等待队列，等待队列会阻塞等待获取锁的线程。在获取锁的时候，会判断当前锁状态变量是否为0，如果为0，则获取到锁，且锁状态变量+1，这时其他锁线程无法获取到锁，当获取锁的同一个线程再次获取锁时，则锁状态变量会再加一。当锁状态变量变为0才会真正释放锁。


什么是AQS？

展开查看

抽象队列同步器，AQS是一个抽象类，用于构建锁和同步器，提供了一些需要实现的通用功能。AQS的实现类有ReentrantLock等。

AQS 使用 int 成员变量 state 表示同步状态(CAS保证原子性)，通过内置的 线程等待队列 来完成获取资源线程的排队工作。state 变量由 volatile 修饰，用于展示当前临界资源的获锁情况。

AQS可以实现公平锁和非公平锁，就看新来的线程是直接竞争锁还是直接去队列排队。


Synchronized 关键字底层是如何实现的？它与 ReentrantLock相比优缺点分别是什么？

展开查看
  • 底层是通过同步代码块实现，1个monitorenter指令指向同步代码块的开始位置，两个monitorexit指向结束位置(正常执行和出现异常两种情况)。执行monitorenter时，会尝试获取对象的锁，如果锁的计数器为 0 则表示锁可以被获取，获取后将锁计数器设为 1 也就是加 1。对象锁的的拥有者线程才可以执行 monitorexit 指令来释放锁。在执行 monitorexit 指令后，将锁计数器设为 0，表明锁被释放，其他线程可以尝试获取锁。
  • 优点：使用简单；synchronized依赖于jvm，可以得到jvm层面的优化； 缺点：功能没有ReentrantLock丰富，比如公平锁，等待可中断

char和varchar 的区别，varchar最多存多大？

事务

说说事务的四大特性(ACID)

展开查看
  • 原子性：指令要么全部成功，要么全部失败；
  • 隔离性：一个用户的事务不会被其他事务干扰；
  • 持久性：数据的改变是持久的，即使数据库发生故障也能恢复；
  • 一致性：执行事务前后，数据保持一致,例如转账业务中，无论事务是否成功，转账者和收款人的总额应该是不变的

其中redo log保证了持久性，undo log保证原子性，隔离性由锁和MVCC共同保证，一致性由其他三个共同保证。

  • MVCC(多版本并发控制可以看作是乐观控制的模式)

MVCC是如何控制读写的？

展开查看

对一份数据会存储多个版本，通过事务的可见性来保证事务能看到自己应该看到的版本。通常会有一个全局的版本分配器来为每一行数据设置版本号，版本号是唯一的。

读操作

  • 当一个事务执行读操作，会先创建快照，使用快照读取，这样其他事务的修改不会影响到当前事务的读取。
  • 若有多个快照版本，选择事务开始前最新的版本

写操作

  • 事务执行写操作时，会生成一个新的数据版本，并将修改后的数据写入数据库
  • 原始版本的数据仍存在，供其他事务快照读取，保证其他事务不受当前事务写操作影响

数据库的事务隔离级别有哪些？各有哪些优缺点？

展开查看 事务的隔离是通过锁和MVCC共同实现的
  • 读取未提交：可能会脏读、不可重复读、幻读
  • 读取已提交：可能会不可重复读和幻读
  • 可重复读：默认的事务隔离级别，幻读还有可能发生
  • 可串行化：所有事务逐个执行

简述脏读、不可重复读和幻读的发生场景

展开查看
  • 脏读：事务读取到其他事务还未提交的数据，如果其他事务回滚了，那读取到的就是脏数据
  • 不可重复读：一个事务多次读取同一数据，得到的结果不一样 (其他事务修改)
  • 幻读：一个事务在读取数据的过程中，突然多了一些原本不存在的数据 (其他事务插入)

如何保证双写一致性

展开查看

对于读数据而言，就是先读缓存，未命中就读数据库并更新缓存。对写数据有两种方法：

和业务挂钩，如果需要强一致性，比如秒杀抢券，加读写锁。如果不加锁，无论是先删缓存还是先操作数据库(缓存过期，A线程读数据库拿到数据后时间片结束，B线程修改数据库后删除缓存，A线程拿着旧数据更新缓存)都会有并发问题。

读写锁分为读锁和写锁，读锁是共享锁，支持其他线程的共享读；写锁是独占锁，阻塞其他线程的读写。

具体用什么读写锁？

  • 如果用Java提供的读写锁，那只能本地，多服务器无法共享
  • 可以用Redission提供的读写锁

第二种方法是异步，适合对数据一致性要求不高的场景，比如缓存帖子内容。更新数据库后，利用消息队列，把更新缓存的操作让消息队列通知消费。

🌟CMS和G1回收器的区别

展开查看 CMS使用标记--清除算法，CMS设计的目的是减少垃圾回收的停顿时间。
  • 初始标记： 暂停所有的其他线程，并记录下直接与 root 相连的对象，速度很快 ；STW
  • 并发标记： 同时开启 GC 和用户线程，用一个闭包结构去记录可达对象。但在这个阶段结束，这个闭包结构并不能保证包含当前所有的可达对象。因为用户线程可能会不断的更新引用域，所以 GC 线程无法保证可达性分析的实时性。所以这个算法里会跟踪记录这些发生引用更新的地方。
  • 重新标记： 重新标记阶段就是为了修正并发标记期间因为用户程序继续运行而导致标记产生变动的那一部分对象的标记记录，这个阶段的停顿时间一般会比初始标记阶段的时间稍长，远远比并发标记阶段时间短。STW
  • 并发清除： 开启用户线程，同时 GC 线程开始对未标记的区域做清扫。

G1划分内存区域后再通过并发标记回收，G1提供可预测的停顿时间，同时兼顾吞吐量和停顿时间。

  • 初始标记
  • 并发标记
  • 最终标记
  • 筛选回收

从 JDK9 开始，G1 垃圾收集器成为了默认的垃圾收集器。

🌟Kafka如何保证消息不丢失

展开查看
  • 生产者发送到brocker丢失 (一般采用异步发送)
    • kafka原生的send方法可以传递callback参数，当发送失败时，会触发callback，可以进行重发等补偿。KafkaTemplate并没有callback，可以使用ListenableFuture监听结果，失败后重试
    • 原生Kafka可以设置重试次数prop.put(ProducerConfig.RETRIES_CONFIG, 10);
  • 消息在brocker存储丢失 (一个主题可以设置多个分区，默认就一个，每个分区有多个副本，一个leader和多个follower，follower中保存副本，leader宕机后补上)
    • 发送确认机制acks。acks=0时，服务器不响应写入成功，速度最快；acks=1(默认)，只要leader收到消息，就向生产者发起成功响应；acks=all，只有全部分区都收到消息，才发成功响应。
  • 消费者从brocker接受丢失
    • kafka实际存储数据的是分区，每个分区有offset，记录已经消费的消息偏移。默认是每5s提交一次offset。如果出现重平衡(负责消费某分区的线程挂了，派个新线程消费)，会重复消费或丢失信息(offset设置好了，消息还没处理就挂了)。可以禁用自动提交，改为手动 (同步提交，异步提交)

什么是僵尸进程和孤儿进程

展开查看
  • 僵尸进程：子进程终止了，但是父进程没有释放子进程占用的资源，导致子进程的PCB仍存在，但无法被使用。父进程需要及时调用 wait()

展开查看

预防

  • 静态分配，直接分配全部资源 (请求和保持)
  • 层次分配 (循环等待)

避免

  • 银行家算法，先试探分配给该进程，通过安全性算法判断分配后系统是否安全，若安全则真正分配

检测

  • 资源分配图，进程用圆圈、资源用方框(方框中一个点代表一份资源)。资源指向进程代表占用，进程指向资源代表申请。需要注意的是，形成环路不一定是死锁。还要看资源是否够用

解除

  • 重启操作系统，结束所有进程
  • 撤销涉及死锁的所有进程
  • 逐个撤销死锁进程，直到死锁解除
  • 抢占资源，从涉及死锁的一个或几个进程中抢占资源，把夺得的资源再分配给涉及死锁的进程直至死锁解除。

tcp三次握手四次挥手

展开查看

三次握手目的是保证双方都有发送和接收的能力，如果只有两次握手，客户端知道自己和服务器的收发没问题，但是服务器只知道客户端的发送和自己的接收没问题。

TCP 的连接状态查看，在 Linux 可以通过 netstat -napt 命令查看。

只要四次挥手没有结束，客户端和服务端就可以继续传输数据！TCP是全双工通信，任何一方都可以在数据传送结束后发出连接释放的通知，待对方确认后进入半关闭状态 (还能接收)。当另一方也没有数据再发送的时候，则发出连接释放通知，对方确认后就完全关闭了 TCP 连接。

服务端的ACK和fin为什么没有合起来发送？ACK代表收到了客户端的断开请求，但这时服务器可能还有东西没发完，等发完再发fin。

但在一些特殊情况下，是允许合并的！如果服务端没有数据要发送，并且tcp默认的延迟确认机制没有关闭，那么就会合并，出现三次挥手，实际在抓包时也会经常出现三次挥手。


TCP延迟确认

展开查看

通常应答报文ACK不携带数据，网络效率较低，因为他也有IP头和TCP头。延迟确认机制就是为了解决这个问题。

  • 当有响应数据要发送时，ACK会随着数据一起立刻发送
  • 当没有响应数据要发送时，ACK 将会延迟一段时间，以等待是否有响应数据可以一起发送
  • 如果在延迟等待发送 ACK 期间，对方的第二个数据报文又到达了，这时就会立刻发送 ACK
